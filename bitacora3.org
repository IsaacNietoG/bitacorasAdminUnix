* Herramientas para máquinas virtuales (Virtual Box y Boxes)

** Conceptos nuevos

*** Virtual Box

Es un software de virtualización desarrollado por Oracle. Mediante este podemos instalar sistemas operativos adicionales "invitados" dentro del sistema "anfitrión".

*** Boxes

GNOME Boxes es una interfaz gráfica que funciona como front-end sobre tecnologías como QEMU y libvirt. Cuando el hardware lo permite, utiliza KVM (Kernel-based Virtual Machine) para acelerar la virtualización. No es un hipervisor distinto, sino una interfaz que facilita el uso de QEMU/KVM.

*** Hipervisor

Es el componente de software encargado de crear, ejecutar y administrar máquinas virtuales. Controla la asignación de CPU, memoria, almacenamiento y dispositivos de entrada/salida para cada sistema invitado.

** Conceptos de repaso

- *Virtualización vs. Emulación*
  - Virtualización: El sistema invitado ejecuta instrucciones directamente sobre la CPU real mediante extensiones como VT-x o AMD-V. Es eficiente y es el modelo más común actualmente.
  - Emulación: Se simula completamente otro hardware o arquitectura distinta (por ejemplo, ARM sobre x86). Implica traducción de instrucciones y es más lenta.

- *Host vs. Guest*
  - Host: Sistema físico donde se ejecuta el hipervisor.
  - Guest: Sistema operativo instalado dentro de la máquina virtual.

- *Snapshot*
  Un snapshot permite guardar el estado de una máquina virtual (disco y en algunos casos, memoria) para regresar a ese punto en caso de error o mala configuración.

- *Imagen ISO vs. Disco Virtual*
  - ISO: Archivo de imagen utilizado como medio de instalación o arranque.
  - Disco virtual (VDI, VMDK, QCOW2): Archivo que actúa como disco duro persistente de la máquina virtual.

** Máximas

- Si no necesitas que la máquina virtual sea visible en la red local, usa NAT.
- El modo puente (bridged) se utiliza cuando la VM debe comportarse como un equipo independiente en la red.
- Antes de realizar cambios importantes en una VM, crea un snapshot.
- Virtualización no implica seguridad automática... solo proporciona aislamiento relativo.
- Si el rendimiento es anormalmente bajo, verifica si se está usando virtualización por hardware o emulación.
- Sin VT-x o AMD-V habilitado en BIOS/UEFI, la virtualización moderna pierde eficiencia o puede no funcionar correctamente.

** Sección libre

En realidad existen dos categorías de hipervisor:

Tipo 1: "Bare Metal" (Nativo)
Este hipervisor se instala directamente sobre el hardware físico. No necesita un sistema operativo previo porque él mismo actúa como un sistema operativo básico diseñado solo para gestionar máquinas virtuales.
Cómo funciona: Tiene acceso directo a los recursos del servidor, lo que lo hace extremadamente eficiente y seguro.
Uso común: Centros de datos, servidores empresariales y la nube (AWS, Azure).
Ejemplos: VMware ESXi, Microsoft Hyper-V (en modo servidor), Xen y KVM.

Tipo 2: "Hosted" (Alojado)
Este hipervisor se instala como una aplicación dentro de un sistema operativo ya existente (como Windows, macOS o Linux).
Cómo funciona: El hipervisor debe pedirle recursos al sistema operativo anfitrión, el cual a su vez se los pide al hardware. Esto añade una capa de "latencia" o sobrecarga, por lo que es menos eficiente que el Tipo 1.
Uso común: Desarrolladores de software, estudiantes o usuarios que necesitan probar sistemas operativos en su propia PC o laptop.
Ejemplos: Oracle VirtualBox, VMware Workstation y Parallels Desktop.

- *Aceleración por hardware*
  Las extensiones VT-x (Intel) y AMD-V (AMD) permiten ejecutar sistemas invitados con menor sobrecarga. Si están deshabilitadas en BIOS/UEFI, el hipervisor puede funcionar en modo limitado o fallar.

- *Posibles conflictos de virtualización*
  En sistemas Linux, los módulos del kernel como kvm_intel o kvm_amd pueden estar cargados. En ciertos casos, VirtualBox puede requerir acceso exclusivo a las extensiones de virtualización, lo que puede implicar descargar temporalmente dichos módulos.

- *Buenas prácticas operativas*
  Nombrar las máquinas virtuales de forma descriptiva (por ejemplo, fedora-lab-2026-02) y documentar los recursos asignados (RAM, CPU, almacenamiento) facilita reproducibilidad y mantenimiento.

* Instalación de virtual box

** Conceptos nuevos

*** Axel

Es un acelerador de descargas que corre desde la terminal. Lo ussamos para acelerar la descarga del ISO de Fedora que luego íbamos a usar posteriormente.

#+begin_src
axel <rutaArchivo>
#+end_src

Internamente utiliza la cabecera HTTP *Range* (HTTP/1.1) para solicitar partes específicas de un archivo. Si el servidor responde con:
''Accept-Ranges: bytes'' entonces soporta descargas segmentadas.

Proceso general:
- Envía una petición HEAD para verificar soporte de rangos.
- Divide el tamaño total del archivo en segmentos.
- Abre múltiples conexiones paralelas solicitando bloques específicos.
- Ensambla los bloques descargados en un solo archivo final.

*** Módulos de kernel

Es una extensión que podemos "pegarle" al kernel para obtener ciertas features particulares. Por ahora, vimos que, si usamos Gnome, entonces tenemos que deshabilitar un módulo de kernel que Boxes instala para acceder a recursos de virtualización directamente al hardware del sistema anfitrión. Para ello, usamos el siguiente comando:

#+begin_src
rmmod kvm_intel
#+end_src

*** Nat y adaptador puente
- Nat(Network Address Translation): Permite que la mv obtenga una ip privada y se conecte a internet utilizando la dirección IP del equipo anfitrión, por lo que la MV tiene acceso al exterior pero no es accesible la red local, en este caso al usar VirtualBox actúa como un router, traduciendo la IP privada de la MV a la pública del host, es muy ideal para la instalación inicial ya que oculta la MV
- Adaptador Puente (Bridged Adapter): Este es la tarjeta de red de la MV que se conecta directamente al adaptador de red del anfitrión, entonces la MV aparecera como un equipo independiente más en la red física, permitiendo la comunicación bidireccional, pero esta es expuesta a ataques externos.

** Conceptos de repaso

- Extensiones de virtualización por hardware
  - VT-x (Intel)
  - AMD-V (AMD)
  Permiten ejecutar múltiples sistemas operativos con menor sobrecarga y mayor eficiencia.

- BIOS/UEFI
  La virtualización por hardware debe estar habilitada desde el firmware del sistema (BIOS/UEFI). Si está desactivada, el hipervisor no podrá usar aceleración.

- Archivo ISO
  Imagen de disco utilizada como medio de instalación del sistema operativo invitado.

- Disco virtual
  Archivo que representa el almacenamiento persistente de la máquina virtual (por ejemplo, VDI en VirtualBox).

** Máximas

- Siempre verifica que VT-x o AMD-V estén habilitados antes de diagnosticar problemas de virtualización.
- Para instalación inicial, usa NAT; cambia a puente solo si el caso lo exige.
- No descargues módulos del kernel sin confirmar que realmente existe conflicto.
- Si una descarga es lenta, confirma que el servidor soporte Accept-Ranges antes de asumir problema de red.
- La virtualización eficiente depende más del hardware que del software.

** Sección libre

Me dió curiosidad investigar cómo funcionaba Axel por detrás y me puse a investigarlo.

Resulta que Axel lo que hace es paralelizar una descarga utilizando varios hilos, cada uno descargando una sección diferente del archivo a la vez. Esto a su vez me llevó a la duda de cómo estaba implementado esto, ya que no es algo que se podría lograr si no estuviera soportado por los servidores de manera nativa. Resulta que es algo que -al menos en el caso de archivos en internet- es soportado directamente por HTTP mediante una feature llamada Byte Ranges

El protocolo HTTP (desde su versión 1.1) incluye una cabecera de solicitud llamada Range. Esta permite al cliente pedirle al servidor solo una parte específica de un recurso, en lugar del archivo completo.

- *Verificación*: Primero, Axel envía una solicitud HEAD para ver si el servidor soporta esto. Si el servidor responde con la cabecera Accept-Ranges: bytes, significa que el servidor es capaz de "trocear" el envío.

- *Segmentación*: Axel divide el tamaño total del archivo (digamos 100 MB) entre el número de conexiones que le pediste (por ejemplo, 4).

- *Descarga en paralelo*: Abre n conexiones simultáneas enviando peticiones como estas:

Conexión 1: Range: bytes=0-24999999

Conexión 2: Range: bytes=25000000-49999999

...y así sucesivamente.

- Seguridad:
  Una VM en modo puente está expuesta a escaneo y tráfico de red como cualquier otro equipo físico. Su configuración de firewall interno sigue siendo necesaria.


* Maquina virtual de fedora

** Conceptos nuevos

*** Spin y flavor

Dentro de fedora existe algo llamada *Fedora Scientific Lab*, que esta construida por una amplia biblioteca de herramientas y software preinstalados diseñados específicamente para la ciencia de datos, el análisis numérico y la computación científica, por lo que obtiene el nombre de "spin". Un spin es una versión alternativa de Fedora que puede cambiar sea en esto o en su entorno de escritorio. Es el caso particular de un flavor.

Un flavor es una versión alternativa de alguna distribución de Linux. Puede ser simplemente un entorno de escritorio distinto por default o toda una serie de paqueterías por default instaladas.

*** Test and start

Por otra parte se menciono que las formas de arranque que nos da Fedora, ya que al iniciar la mv nos da la opción de *"Test this media & start Fedora"* y sucede que es muy recomendable seleccionarla ya que asegura la integridad de la imagen ISO descargada, ya que como dato curioso, el profesor nos menciono que anteriorme se arrancaba desde CDs/DVDs físicos y solian estar dañados.

*** Terminales

- MV: Esta la encontraremos de color negro ya que indica que estás operando dentro del sistema operativo invitado ("guest OS"), aislado y ejecutándose sobre un hipervisor.

- Dependiendo de cada institución se pueden agregar más elementos, incluso paquetes de otras distribuciones obteniendo una similitud en apariencia pero aun estando en fedora.

** Conceptos de repaso

*** Nat y adaptador puente
- Nat(Network Address Translation): Permite que la mv obtenga una ip privada y se conecte a internet utilizando la dirección IP del equipo anfitrión, por lo que la MV tiene acceso al exterior pero no es accesible la red local, en este caso al usar VirtualBox actúa como un router, traduciendo la IP privada de la MV a la pública del host, es muy ideal para la instalación inicial ya que oculta la MV
- Adaptador Puente (Bridged Adapter): Este es la tarjeta de red de la MV que se conecta directamente al adaptador de red del anfitrión, entonces la MV aparecera como un equipo independiente más en la red física, permitiendo la comunicación bidireccional, pero esta es expuesta a ataques externos.

Esto lo usamos para configurar la VM de Fedora.

- Imagen Live vs Instalación completa
  - Live: entorno temporal para pruebas.
  - Instalación completa: sistema persistente en disco virtual.

- Espacio requerido
  Fedora Workstation requiere aproximadamente 3.8 GB como mínimo, aunque en práctica se recomienda asignar más espacio para actualizaciones y software adicional.

- NAT vs. Adaptador puente
  - NAT: VM aislada y protegida detrás del host.
  - Puente: VM visible como equipo independiente en la red.

- Terminal del host vs terminal de la VM
  La terminal del host ejecuta comandos en el sistema físico.  
  La terminal dentro de la VM ejecuta comandos aislados en el sistema invitado.

** Maximas

- Siempre verificar la integridad de la ISO antes de instalar (el checksum).
- Una imagen Live no sustituye una instalación persistente.
- No habilitar repositorios de terceros sin conocer su procedencia.
- Asignar espacio adicional al mínimo recomendado evita problemas posteriores.
- Diferenciar claramente cuándo se está en el host y cuándo en la VM previene errores críticos.

** Sección libre

- Buenas prácticas durante instalación:
  - Asignar al menos 2 GB de RAM para entorno gráfico fluido.
  - Crear usuario no root para uso cotidiano.
  - Actualizar el sistema inmediatamente después de instalar:
    
    #+begin_src
    sudo dnf update
    #+end_src

- Aislamiento:
  Aunque la VM está aislada del host, comparte hardware físico. Un uso excesivo de CPU o memoria en la VM impacta directamente el rendimiento del host.

- Arquitectura:
  La ISO debe corresponder a la arquitectura del procesador.

* Lectura de archivos y comandos

** Conceptos nuevos
#+begin_src
lss nfc-blacklist.conf
#+end_src
Este se utiliza a menudo para impedir que los drivers automáticos del kernel se carguen, lo cual es muy importante ya que evita conflictos con herramientas de terceros como *libnfc* o *pcscd*

#+begin_src
lss firewall-systems.conf
#+end_src
En este archivo es una configuración de firewalls, que se basa en archivos de configuración dependiendo de la herramienta de red utilizada, se definen zonas de red, servicios permitidos, puertos y protocolos.

#+begin_src
ls /etc/mtad /etc/fstab
#+end_src
Estos archivos gestionan el montaje de sistemas de archivos.
- *fstab* (file systems table) es un archivo estático y persistente configurado por el usuario para montar dispositivos al arrancar, asi como las particiones, discos y recursos compartidos al iniciar. (configuración)
- *mtab* (mounted filesystems table) es un archivo dinámico actualizado por el sistema que enumera los sistemas de archivos montados actualmente. (estado)


** Conceptos de repaso
#+begin_src
rmmod kvm_intel
#+end_src
El comando *rmmod* se utiliza para eliminar un módulo del kernel de linux, mientras que *kvm_intel* es un módulo del kernel específico para procesadores intel que permite utilizar las extensiones de virtualización por hardware para KVM.

#+begin_src
top
#+end_src
Este comando permite tener una vista dinámica y en tiempo real de los procesos activos, ordenándolos por defecto según el uso de CPU, aquí le prestamos más atención al apartado de *load average*, ya que muestra la carga promedio del sistema, donde indican cuántos procesos están ejecutándose o esperando tiempo de CPU, esto siendo crucial con el número de núcleos del servidor, ya que cada núcleo solo puede trabajar con un proceso.

** Máximas

- Antes de modificar un archivo en /etc, respáldalo.
- fstab mal configurado puede impedir que el sistema arranque.
- El load average debe interpretarse respecto al número de núcleos.
- No descargues módulos del kernel sin verificar dependencias.
- Firewall activo no implica configuración segura; depende de reglas definidas.

** Sección libre

- Buenas prácticas al leer archivos de configuración:
  - Usar less en lugar de cat para archivos largos:
    
    #+begin_src
    less /etc/fstab
    #+end_src


- Sobre /proc:
  Muchos estados dinámicos del sistema no están en archivos físicos, sino en sistemas virtuales como /proc.  

  #+begin_src
  cat /proc/cpuinfo
  #+end_src

  Esto muestra información del procesador directamente expuesta por el kernel.

- Seguridad:
  Archivos dentro de /etc definen comportamiento crítico del sistema.  
  Modificaciones incorrectas pueden comprometer estabilidad o seguridad.

* Directorios

** Conceptos nuevos

*** /proc
Es un sistema de archivos virtual (procfs) generado dinámicamente por el kernel y montado en memoria al iniciar el sistema.  
No existe físicamente en el disco.

Contiene información en tiempo real sobre:
- Procesos (/proc/<PID>/)
- CPU (/proc/cpuinfo)
- Memoria (/proc/meminfo)
- Parámetros del kernel (/proc/sys/)

#+begin_src
cat /proc/meminfo
#+end_src

Su contenido cambia constantemente porque refleja el estado actual del sistema.

*** /dev
Directorio que contiene archivos especiales que representan dispositivos.  
En Unix/Linux, los dispositivos se modelan como archivos.

Ejemplos:
- /dev/sda → disco duro
- /dev/tty → terminal
- /dev/null → descarta toda salida
- /dev/zero → genera flujo continuo de ceros

Estos archivos no almacenan datos; actúan como interfaz hacia el hardware o dispositivos virtuales.

*** /etc
Contiene archivos de configuración del sistema.  
No guarda binarios ni datos de usuario, sino parámetros de funcionamiento.

Ejemplos:
- /etc/fstab
- /etc/passwd
- /etc/hosts

Es uno de los directorios más críticos del sistema.

*** /var
Almacena datos variables que cambian durante la ejecución del sistema.

Incluye:
- Logs (/var/log)
- Caché (/var/cache)
- Archivos temporales persistentes
- Spool de impresión

A diferencia de /etc su contenido es dinámico y crece con el uso.

** Conceptos de repaso

*** Sistema de archivos
Es la estructura lógica que organiza cómo se almacenan y recuperan datos en un dispositivo.

Ejemplos:
- ext4
- BTRFS
- XFS

*** BTRFS
Sistema de archivos moderno con características avanzadas como:
- Snapshots
- Compresión
- Subvolúmenes
- Verificación de integridad

*** Montaje de sistemas de archivos
El proceso de asociar un sistema de archivos a un directorio dentro del árbol principal (/).

#+begin_src
mount /dev/sdb1 /mnt
#+end_src

Todo en Linux se integra bajo un único árbol jerárquico que comienza en / (raíz).

** Máximas

- En GNU/Linux, todo es un archivo (incluyendo dispositivos).
- /proc y /dev no almacenan datos reales en disco.
- Modificar archivos en /etc impacta el comportamiento global del sistema.
- El sistema de archivos comienza en /, no existen “letras de unidad”.
- Montar incorrectamente un sistema puede ocultar el contenido previo del directorio.

** Sección libre

- Estructura jerárquica:
  GNU/Linux no usa particiones visibles como C:, D:.  
  Todo se organiza bajo una sola raíz (/), y otros discos se montan dentro de esa jerarquía.

- IMPORTANTE:
  Antes de manipular discos o particiones se debe verificar con

  #+begin_src
  lsblk
  #+end_src

  Esto muestra dispositivos y puntos de montaje actuales.

- Seguridad y permisos:
  Cada directorio posee permisos (rwx) para usuario, grupo y otros.  
  Una mala configuración en directorios críticos puede exponer información sensible o bloquear servicios.
